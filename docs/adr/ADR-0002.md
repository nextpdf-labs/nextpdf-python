# ADR-0002: pypdf as Internal Dependency for PDF Structure Reading

**Status**: Accepted  
**Date**: 2026-04-01

## Context

The local extraction engine (ADR-0001, Route B) requires the ability to parse
PDF internal structures -- specifically the StructTree (tagged PDF structure) and
page content streams -- from pure Python without C extensions.

Key constraints:

1. **Pure Python required.** The SDK targets agent sandboxes and ephemeral
   containers where C compilation toolchains are unavailable. Binary wheels
   with C extensions create portability and supply-chain risks.
2. **BSD-compatible license required.** The SDK is commercially distributed.
   Copyleft or viral licenses (LGPL, GPL) are not acceptable for a dependency
   that is linked at the library level.
3. **StructTree traversal is the critical path.** The local engine must read
   `/StructTreeRoot`, walk `/K` arrays, resolve `/MCID` references, and map
   structure elements to page content. This is the minimum viable capability.

## Decision

Use **pypdf** (BSD-3-Clause, pure Python) as the internal dependency for PDF
structure reading.

pypdf provides:

- Low-level access to PDF objects, cross-reference tables, and the object tree.
- Page content stream access for text extraction.
- Pure Python implementation with no C extension requirements.
- Active maintenance with regular releases and broad community adoption.
- BSD-3-Clause license fully compatible with commercial distribution.

pypdf is declared as a **runtime dependency** in `pyproject.toml` and used
exclusively through internal modules (not re-exported in the public API).

**Fallback plan:** If pypdf's object-level API proves inadequate for StructTree
traversal (e.g., cannot resolve indirect references in `/K` arrays, cannot
handle cross-reference streams in modern PDFs), fall back to a **thin
zlib+struct reader** scoped exclusively to StructTree traversal. This reader
would parse only the minimal PDF structures needed (xref table, object streams,
StructTree dictionaries) without attempting general-purpose PDF parsing.

## Consequences

**Positive:**
- Avoids months of effort building a PDF parser from scratch.
- pypdf is battle-tested on a wide range of PDF variants.
- BSD license eliminates legal review overhead.
- Pure Python ensures portability across all target environments.
- Active upstream means security patches flow without SDK team effort.

**Negative:**
- Adds ~2 MB to installed package size.
- SDK inherits any pypdf bugs in object resolution or stream decoding.
- pypdf's StructTree support is not its primary focus; may need workarounds.
- If fallback reader is needed, it represents additional maintenance surface.

## Alternatives Rejected

- **pikepdf (LGPL-3.0 + QPDF C dependency):** LGPL creates viral licensing
  risk when linked as a library dependency. QPDF C dependency violates the
  pure-Python constraint. Despite superior PDF manipulation capabilities,
  the licensing and portability risks are disqualifying.
- **Building a PDF parser from scratch:** The PDF specification (ISO 32000-2)
  is ~1,000 pages. Implementing even the subset needed for StructTree
  traversal (cross-reference tables, object streams, encryption, content
  streams) represents XL engineering effort with high defect risk. Not
  justified when pypdf provides 80%+ of the needed capability.
- **pdfminer.six:** Focuses on text extraction rather than structure tree
  access. Does not expose the low-level PDF object tree needed for StructTree
  traversal. Also has unclear maintenance status.
- **PyMuPDF (fitz):** AGPL license is incompatible with commercial
  distribution. Also requires C extensions (MuPDF).

## Revisit Trigger

- pypdf cannot extract StructTree data from >= 80% of tagged PDFs in the test
  corpus (triggers fallback to thin zlib+struct reader).
- pypdf changes its license to something incompatible with commercial use.
- A new pure-Python, BSD-licensed library emerges with superior StructTree
  support.
