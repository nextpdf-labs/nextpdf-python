# ADR-0001: Route B Strategic Direction -- Local-First with Fallback Ladder

**Status**: Accepted  
**Date**: 2026-04-01

## Context

The nextpdf-python SDK is currently a thin HTTP client (~700 LOC) that delegates
all PDF extraction work to a remote nextpdf-server instance. This creates a hard
runtime dependency on network access and a running server, which blocks two
high-value use cases:

1. **Agent sandboxes** -- AI agents running in ephemeral containers with no
   outbound network access need local PDF extraction.
2. **Edge / offline deployments** -- CLI users and CI pipelines benefit from
   zero-infrastructure PDF extraction.

Route A (server-only SDK) preserves the current architecture but cannot serve
these use cases. Route B (local-first with server fallback) adds a pure-Python
extraction engine inside the SDK itself, making the server optional rather than
required.

## Decision

Adopt **Route B: local-first architecture** with a three-stage fallback ladder
for the GA release:

| Stage | Trigger | Description |
|-------|---------|-------------|
| **S1** | Local engine passes quality gate | GA ships with full local engine as the default extraction path. Server becomes an optional fallback for advanced features (e.g., OCR, scanned PDFs). |
| **S2** | Local engine partially passes gate | GA ships the SDK (remote client) as stable. Local engine ships as a beta feature behind `nextpdf extract --local` / `engine="local"` flag. |
| **S3** | Local engine fails quality gate | GA ships SDK-only (remote client). Local engine deferred to next minor release. |

**Default recommendation: S2.** Promote to S1 only if the local engine spike
passes the quality gate defined in the GA test plan (F1 >= 0.95 on tagged corpus,
IoU >= 0.90 on bounding boxes, zero data-loss regressions on the canary set).

The fallback ladder is evaluated once at the end of the local-engine spike phase.
The decision is recorded as a follow-up ADR.

## Consequences

**Positive:**
- SDK gains standalone value without infrastructure dependency.
- Agent sandbox and offline use cases are unblocked.
- Server remains available for heavyweight operations (OCR, large-batch).
- S2/S3 stages provide safe fallback if local quality is insufficient.

**Negative:**
- Engineering investment in local engine with uncertain quality outcome.
- Increased package surface area and maintenance burden.
- Must maintain two extraction paths (local + remote) long-term.
- pypdf becomes a required dependency, increasing install footprint.

## Alternatives Rejected

- **Route A (server-only SDK):** Does not address agent sandbox or offline
  requirements. Rejected because the primary growth vector is agent tooling,
  which demands local execution.
- **Hybrid-only (always require server for StructTree):** Reduces local engine
  scope but still requires network access, defeating the purpose.

## Revisit Trigger

- Local engine spike fails the quality gate entirely (triggers S3).
- A superior pure-Python PDF extraction library emerges that replaces the
  custom engine.
- Market analysis shows agent sandbox use case is not a priority (unlikely
  given current trajectory).
