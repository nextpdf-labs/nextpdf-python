# ADR-0003: CLI Framework Selection -- click

**Status**: Accepted  
**Date**: 2026-04-01

## Context

The nextpdf-python SDK requires a CLI entrypoint (`nextpdf` command) for
direct terminal usage. The CLI will expose subcommands for PDF extraction,
AST inspection, and server connectivity checks. The framework must support:

1. **Subcommand groups** -- e.g., `nextpdf extract`, `nextpdf ast show`,
   `nextpdf server ping`.
2. **File and stdin input** -- PDF paths and piped input.
3. **Structured output** -- JSON output mode for programmatic consumption.
4. **Minimal dependency footprint** -- the SDK is used as a library dependency
   in agent frameworks; transitive dependency bloat is a real cost.
5. **Python 3.10+ compatibility** -- must work across the supported version
   range.

## Decision

Use **click** as the CLI framework.

click provides:

- Mature, well-documented subcommand and group system.
- Decorator-based command definitions that are readable and maintainable.
- Built-in support for file types, stdin/stdout, and parameter validation.
- Single dependency (no transitive pulls beyond click itself).
- BSD-3-Clause license, compatible with commercial distribution.
- Established ecosystem with wide adoption (Flask, Black, pip all use click).

click is declared as a runtime dependency in `pyproject.toml` and used
exclusively in the `nextpdf.cli` module. The CLI entrypoint is registered
as a console script: `nextpdf = nextpdf.cli:main`.

## Consequences

**Positive:**
- Single, well-maintained dependency with no transitive bloat.
- Decorator-based API produces self-documenting command definitions.
- Subcommand groups map naturally to the SDK's domain structure.
- Extensive documentation and community knowledge base.
- Compatible with lazy imports -- CLI module is only loaded when invoked.

**Negative:**
- click is synchronous by default; async commands require wrapping with
  `asyncio.run()` or `anyio.from_thread.run`. This is acceptable since
  CLI commands are inherently sequential.
- No built-in rich output formatting (tables, progress bars with color).
  Can be added later with `rich` as an optional dependency if needed.
- Type inference from decorators is less precise than typer's approach,
  requiring explicit `@click.option` annotations.

## Alternatives Rejected

- **typer:** Built on top of click but pulls in `rich` and
  `typing-extensions` as transitive dependencies. The additional ~5 MB
  install footprint is not justified for the marginal ergonomic improvement.
  typer's type-inference approach also complicates custom parameter types
  and group-level options. If the project already depended on rich, typer
  would be reconsidered.
- **argparse (stdlib):** Zero-dependency but significantly less ergonomic
  for subcommand groups. No decorator-based API, manual help formatting,
  and verbose boilerplate for nested subcommands. Maintenance cost is higher
  for the same feature set.
- **fire:** Automatic CLI generation from functions/classes. Too implicit
  for a public-facing CLI where explicit parameter definitions, help text,
  and validation are required. Also lacks subcommand group support.

## Revisit Trigger

- click proves insufficient for async command support and the workaround
  (`asyncio.run()` wrapper) creates user-facing issues.
- The SDK adopts `rich` as a dependency for other reasons, making typer's
  transitive cost zero.
- click maintenance stalls or a security issue goes unpatched.
